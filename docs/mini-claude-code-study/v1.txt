v0证明了"一个bash工具就够用"，但也留下了明显的隐患——所有操作都靠模型自己拼shell字符串，没有任何结构化的输入校验，路径穿越、危险命令这些风险完全没设防。v1的核心改动就是从"一个通用接口"换成"四个专用工具"：bash、read_file、write_file、edit_file。
每个工具都有自己精确的input_schema，比如edit_file要求三个字段：path、old_text、new_text。这比v0那种"只有一个command字符串字段、剩下全靠模型自己组织"的方式更可控——模型被约束着必须传结构化的参数，而不是自由发挥拼命令行。
对应的execute_tool函数是个很典型的dispatch table模式：根据name路由到不同的run_xxx函数。这个模式后面v2、v3、v4一直在用，扩展一个新工具只需要三步——加schema、加分支、写实现函数，互不干扰。这个写法我觉得挺值得记住，以后自己写工具集成的时候会用到。
路径逃逸防护（safe_path函数）：把传入路径和工作目录拼接后用.resolve()解析成绝对路径，再检查这个真实路径有没有还落在工作目录范围内。这个检查很必要——如果模型（或者被注入的恶意内容）传入类似../../etc/passwd这种路径，.resolve()会把..解析掉，暴露出它实际指向的位置，这时候拦下来正合适。这是"最小权限原则"在代码层面的体现：不管模型主观上想不想干坏事，权限本身就不该给到能碰工作目录之外文件的程度。
危险命令黑名单（对bash的过滤）：简单的字符串匹配，防护力有限（多打个空格都能绕开），但它体现了一个态度——即便有了专用工具，bash依然是最危险的入口，不能放任不管。
edit_file的设计，content.replace(old_text, new_text, 1)——第三个参数1限定只替换第一次出现的匹配。这个细节我觉得是这一版里设计感最强的地方：模型必须传入足够独特的文本片段才能保证改中的是它真正想改的位置，如果old_text在文件里没找到，直接报错让模型自己调整重试，而不是静默失败或者改错地方。这跟我们对话过程中，Claude实际编辑代码时用的str_replace工具是同一个思路——必须唯一匹配才允许替换，防止误改。
v0只在最外层处理过一次subprocess超时，v1里每个run_xxx函数都自己包了try/except，把异常转换成字符串返回，而不是任由异常网上抛导致整个程序崩溃。这样设计的好处很直接：任何一次工具调用失败，都不会打断整个agent循环，模型能看到"这次操作失败了，原因是XXX"，然后自己决定要不要调整策略重试。SYSTEM prompt里那句"Never invent file paths. Use bash ls/find first if unsure"正好是配合这套机制的——工程实现和prompt指导是配套设计的，不是各干各的。
举一个例子，拿"把config.py里DEBUG=True改成False"这个任务走了一遍完整流程，印象最深的是模型的行为模式变了：v0里模型大概率会直接拼一条sed -i命令一次搞定，v1里因为有了专用的read_file+edit_file，模型明显更倾向于"先读确认、再精确编辑"这种两步走。这说明工具本身的存在会引导模型的操作习惯，不是单纯多了几个接口这么简单——给它专用工具，它会更倾向用更谨慎的方式完成任务，这是我之前没有想到的一层。
另外args.get("limit")和args["limit"]的区别也是这次才真正搞清楚：前者查不到key时返回None不报错，后者会直接抛KeyError。工具参数里那些非必填字段，都得用.get()取，这个小细节容易在写代码时忽略。
v0到v1这一步，本质上是从"相信模型能靠通用工具搞定一切"转向"给模型划定安全边界、用结构化接口降低犯错空间"。这跟我们后面聊到的真实Claude Code的沙箱机制其实是同一个方向上的早期形态——只是v1用的是应用层的路径检查和字符串黑名单，真实产品用的是操作系统级别的虚拟机隔离，防护强度差了好几个量级，但思路的出发点是一致的：不能完全指望模型自我约束，工程上要有兜底。
