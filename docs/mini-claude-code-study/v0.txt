整份代码在做的事，压缩成一句话就是：
while True:
    response = 模型看着history生成
    if 模型不想再调工具了: return
    执行模型要求的工具调用
    把执行结果塞回history
chat()函数里那个while True就是这个loop的具体实现。我一开始容易漏掉的一点是：不管这一轮模型返回的是纯文本还是工具调用，history.append({"role": "assistant", ...})这一步都必须做，不然模型就没法在下一轮"记得"自己刚才决定调用了什么。
stop_reason这个字段一开始我以为是代码里手动赋值的，后来搞清楚了——这是API服务端返回的，不是本地代码设的值，代码里只是拿它做if判断（!= "tool_use"就结束循环）。这也顺带让我理解了一个更本质的问题：模型的输出格式（工具调用要不要是合法JSON、参数对不对）根本不是靠SYSTEM prompt里写文字去约束的，而是tools=TOOL这个独立参数在起作用——这是API协议层+ 模型专项训练的能力，跟prompt文本完全解耦。以前我一直以为格式规范全靠prompt里写"请输出JSON"这种话去哄模型，现在明白tool calling这个机制本身就是把"说什么"和"怎么说"拆成了两个通道。
v0的子代理机制很巧妙
v0没有真正意义上的"多agent架构"，子代理就是靠subprocess.run开一个全新的Python进程去跑同一份脚本：
pythonif len(sys.argv) > 1:
    print(chat(sys.argv[1]))   # 子代理模式，全新的空history
主agent觉得某个任务该独立处理，就在工具调用里生成一条形如python v0_bash_agent.py '任务描述'的shell命令，父进程的subprocess.run会真的开一个子进程去跑，子进程从sys.argv[1]里拿到任务描述，用全新的、空白的history（因为chat(sys.argv[1])没传第二个参数，用的是函数签名里history=None触发的默认空列表）处理完，print出结果，父进程通过out.stdout把这段打印内容捕获回来，当作这次工具调用的返回值。
这个设计让我印象比较深的地方是：上下文隔离不是靠代码里维护什么复杂的多分支数据结构实现的，而是直接借用了操作系统的进程隔离——开一个新进程，内存天然互不干扰，跑完销毁，干净利落。虽然v0里没细讲，但这个思路后来在v3被替换成了同进程内维护独立sub_messages列表的方式，两种做法的核心目的一样（互不污染上下文），实现代价不同。
从"教学简化版"到"能跑的完整版"之间差在哪
还有一些需要注意的细节
配置不是硬编码而是env+环境变量（load_dotenv()、os.getenv(...)），避免密钥直接写死在代码里
assistant消息存回history前做了显式的字典重建，不是直接塞SDK返回的对象。这样history理论上能被json.dumps()序列化落盘，SDK自带的类实例不一定能被默认序列化器处理
超时用try/except接住，不会让整个程序在命令跑太久时直接崩溃，而是把"超时了"当作一次正常的工具执行结果反馈给模型，让它自己判断要不要换个思路
输出做了[:50000]长度截断，防止一次意外的大量输出把上下文直接撑爆
交互退出加了EOFError/KeyboardInterrupt的兜底，Ctrl+C不会让程序难看地崩溃报错
v0最想让人记住的可能不是具体的代码技巧，而是它想证明的那句话：agent不需要精心设计的专用工具体系才能工作，一个足够通用的执行接口（bash）+ 一个循环，配合模型本身在训练阶段积累的"知道怎么拼shell命令"的能力，就已经是一个完整的agent了。后面v1到v4加的所有东西——专用工具、安全护栏、todo管理、子代理权限隔离、技能按需加载——都是在这个骨架基础上，为了让它在真实场景里更安全、更可控、更适合长任务而逐步叠上去的工程量，不是对这个核心循环本身的颠覆。
