v3解决的是"怎么把任务拆给合适的执行者"，v4解决的是另一个方向的问题：怎么让执行者（不管是主agent还是子agent）临时具备某个专业领域的详细操作知识，而不需要一开始就把这些知识全塞进SYSTEM prompt里占位置。核心新增是SkillLoader类，配合SKILL.md这套文件标准。
SkillLoader是预加载摘要，按需加载全文，这一版最核心的设计分成两个阶段，我觉得是整套教程里最精巧的一处。
第一阶段（程序启动时，__init__里立即执行）：load_skills()扫描skills/目录下每个子文件夹里的SKILL.md，用正则把YAML frontmatter和正文body拆开，解析出name、description存进内存里的self.skills字典。这一步只读元数据（名字和一句话描述），不涉及正文全文，开销很小，而且是一次性的，跟后面用户问什么完全无关。
第二阶段（真正被Skill工具调用时才执行）：get_skill_content()才会把完整的body正文拼出来，还会扫描这个技能目录下有没有配套的scripts/、references/、assets/子文件夹，把里面的文件名也罗列进去。这一步是惰性的,只有模型真的决定要用某个技能时才发生。
这个"预加载摘要 + 按需加载全文"的两段式设计，直接对应了SYSTEM prompt里只放得下所有技能的简短描述这个现实约束——真正详细的操作指南可能很长，不会一开始就塞进上下文，只有被点名调用时才通过tool_result注入进对话。这跟我自己在做的Habsburg RAG项目其实是同一类问题的不同解法：我的项目是"检索相关文档片段，按需塞进prompt"，这里是"检索相关技能文件，按需塞进对话历史"，本质上都是在解决"有限的上下文窗口，怎么在恰当的时机装进恰当的信息"这个核心矛盾。这个类比是我们对话里聊到的，我觉得挺能帮我把这版代码和自己手头的项目对上号。
re.match(r"^---\s*\n(.*?)\n---\s*\n(.*)$", content, re.DOTALL)
这条正则一开始没完全看懂re.DOTALL的作用——默认情况下正则里的.不匹配换行符，但技能正文必然是多行的Markdown文本，不加这个flag的话(.*)$根本抓不到多行内容。这算是这次学到的一个正则表达式的实际应用坑点。
frontmatter部分没有用真正的YAML解析库，而是手写了个极简版本，靠line.split(":", 1)（限制只分割一次）来处理key: value格式，这样即便value本身包含冒号也不会被错误地多切一刀。.strip("\"'")再去掉value首尾可能带的引号。这种"够用就好、不引入额外依赖"的手写解析思路，跟整个教程一直坚持的"能不加复杂度就不加"的教学取向是一致的。
run_skill的返回格式：自定义标签包裹
f"""<skill-loaded name="{skill_name}">
{content}
</skill-loaded>
Follow the instructions in the skill above to complete the user's task."""
这不是Anthropic API强制要求的格式，是这份代码自己约定的——用一对显眼的标签把"刚被动态注入进来的技能内容"和对话里其他普通文本区分开，模型看到这种标签包裹的内容会更容易识别出这是一段结构化的指令补充,而不是把它当成用户随口说的话。末尾那句明确的行动指令（"照着上面的说明做"）也是刻意加的,不是单纯注入内容就完事,还得告诉模型接下来该干嘛。
权限收窄的思路又延续了一次
SKILL_TOOL只加进了主agent的ALL_TOOLS，没有加进get_tools_for_agent筛选后的子代理工具集——回顾v3的get_tools_for_agent只从BASE_TOOLS里筛，Task和Skill都不在BASE_TOOLS里，所以子代理既不能再spawn新的子代理，也不能加载Skill，这两个"高级能力"都只保留给主agent。这是v3那套"限制子代理权限范围"思路的直接延续，说明这几版之间不是各自独立设计的，核心哲学是一路贯穿下来的。
拿"提取PDF文字内容"这个任务走了一遍。最直观的感受是：在模型第一次调用Skill工具之前，它对这个技能的了解仅限于SYSTEM prompt里那一行"- pdf-processing: Extract text and tables from PDF files"，直到run_skill真正执行、get_skill_content现场拼装出完整正文，这段详细操作说明才第一次进入对话上下文——这就是"按需加载"最字面的意义,之前只是听这个词,这次是亲眼看着它在history里从"没有"变成"有"的那个具体时刻。
另一个印象深的点是：同一次对话里Skill工具通常只会被调用一次。技能内容一旦进了history，就跟其他任何历史消息一样永久留在对话记录里，模型不需要每一轮都重新加载。如果后面又要处理第二份PDF，模型完全可以直接调用bash跑脚本，不用重新触发Skill——因为它在这次对话的记忆里已经"学会"了。这也解释了为什么要设计成"一次注入、终身受用于这次对话"而不是每次都重新加载,不然会重复消耗上下文空间去装同一份说明书。
一点自己的思考，串联v0到v4
这版看完之后回头梳理五个版本，会发现一条很清楚的主线——每一版都在用不同的手法回答同一个问题：这一刻，模型的上下文窗口里，该装什么、不该装什么。
v0：证明"给模型一个通用接口+一个循环"就是agent的全部本质
v1：用专用工具收窄"能做什么"，把裸露的执行能力换成结构化的、带安全边界的接口
v2：用显式Todo管理解决"记不记得该做什么"，把计划从模型脑子里的隐式想法变成对话历史里能被反复读到的文本
v3：横向拆分"谁该看见什么"，用同进程内独立的sub_messages实现上下文隔离,让探索性的脏活不污染主线
v4：纵向按需加载"什么时候该装进什么"，专业知识平时只占一行摘要,真正用到时才临时膨胀注入
这五版加起来做的事,说到底都是在给"模型是决策者,代码只是执行者"这句话找配套的工程实现——代码没法替模型做判断,但可以决定模型每一轮能看见什么、被允许调用什么、什么时候该被提醒。这大概是整个教程读下来,对我来说最有迁移价值的一条认知。
